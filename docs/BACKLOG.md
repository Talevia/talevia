# Backlog

`/iterate-gap` 的驱动源。每个 cycle 的流程：

1. 读这个文件，挑**最靠上的未完成**任务（P0 → P1 → P2，同档内按出现顺序）。
2. plan → 实现 → 验证 → 归档决策 → commit + push。
3. 在 `docs(decisions)` 那次 commit 里顺手把本任务的 bullet 从这个文件里删掉（不是打钩，直接删），保持列表紧凑。
4. 如果这个文件的 P0/P1/P2 三档都已清空（或文件不存在），`/iterate-gap` 会按 skill 里的 rubric 分析一次性生成 **20 条新任务**写回这个文件、commit，然后**在同一个 cycle 里继续**挑新生成列表里第 1 条开干。

任务描述只给 **Gap（现状 / 痛点）** 和 **方向（期望动的东西）** —— 细节留给 cycle 里的 plan 阶段现读代码决定。Rubric 轴用 `§5.x` 标。

## P1 — 中优，做完 P0 再排

- **debt-split-project-maintenance-action-tool** — `core/tool/builtin/project/ProjectMaintenanceActionTool.kt` is 526 lines (just over R.5 #4's 500 threshold). Sixth consolidation landed three actions (prune-lockfile / gc-lockfile / gc-render-cache) in one file; natural split axis is one file per action handler with a shared dispatch + PermissionSpec block. **方向：** extract `PruneLockfileHandler` / `GcLockfileHandler` / `GcRenderCacheHandler` siblings; main tool class becomes ≤ 150 lines of dispatch. Axis: "new maintenance action" (each new action pressures the same file). Rubric §5.6.
- **export-incremental-render-phase-1-cache-key-design** — first of the three-split the 2026-04-23 skip-tag requested. **方向：** decision-only cycle — define `ExportTool`'s per-clip cache-key composition precisely: `(clip spec hash × source binding deep-hash × render profile hash × engine-id)`. Enumerate which mutations must invalidate which clips (e.g. changing `Timeline.duration` must NOT invalidate unaffected clips). No code changes yet; phase 2 + 3 bullets depend on this cycle's decision file. Rubric §5.7.
- **export-incremental-render-phase-2-per-clip-memoization** — **方向：** wire `ClipRenderCache` to the key from phase 1; `ExportTool` looks up per-clip artefacts and only dispatches `renderClip` for keys not found. Benchmarks (cycle 18 baseline 0.913 ms/op whole-timeline) must show linear improvement with reused-clip ratio. **前置：** phase 1 decision must land first. Rubric §5.7.
- **export-incremental-render-phase-3-invalidation-edge-tests** — **方向：** counter-intuitive edge tests (§3a #9): (a) same clip on two tracks — changing one track order must not invalidate the other's cache entry; (b) identical clip spec, different source DAG ancestors — must miss; (c) cross-engine cache re-use — FFmpeg-rendered artefact must NOT serve a Media3 request. **前置：** phase 2 must land first. Rubric §5.7.
- **debt-cross-container-tool-list-builder** — 4 AppContainers (CLI / Desktop / Server / Android) keep near-identical `register(...)` blocks in sync by convention; any new tool needs 4 import-line insertions + 4 register-line insertions + 4 ktlint-sort passes. Drift bugs show up as "tool missing on platform X". **方向：** expose `core.tool.builtin.defaultBuiltinRegistrations(...)` that takes the concrete dependencies (stores / engines / provider-auth / etc.) and returns the shared list of `RegisteredTool`; each container `apply`s it then overlays platform-specific tools (e.g. AIGC tools gated on keys). One SSoT, zero drift. Rubric §5.6.
- **debt-filestore-benchmark** — `core/domain/FileProjectStore.kt` is the hottest-path store (every tool mutation + every session load); zero wall-time benchmark. On large projects the `openAt` (deserialize talevia.json) + `mutate` (re-serialize) costs are load-bearing but unguarded. **方向：** add `FileProjectStoreBenchmark` to `:benchmark` covering `openAt(large bundle)`, `upsert(N-clip project)`, `list()`. Record baseline numbers in the decision file. Rubric §5.7.
- **core-provider-cost-compare-query** — `provider_query` has 2 selects today (cycle 29 scan); no way for the agent to answer "which model is cheapest for this 10k-token request?". **方向：** add `provider_query(select=cost_compare)` returning per-(providerId, modelId) cents-per-1k-input + cents-per-1k-output from a static pricing table in `core/provider/pricing/`. Input: requestedInputTokens / requestedOutputTokens; output: sorted rows. Matches the VISION §5.2 "generate_image vs dall-e-3" tradeoff the agent is expected to navigate. Rubric §5.2.
- **compaction-diagnosis-query** — cycle 26 added `BusEvent.SessionCompacted(prunedCount, summaryLength)`; ops can see "it compacted" live, but can't audit historical compactions on a re-opened session. **方向：** `session_query(select=compaction_trace)` enumerates `Part.Compaction` entries in session order with `(replacedFromMessageId, replacedToMessageId, createdAt, summaryPreview)`. Lets the agent + operator reconstruct what each compaction pass dropped. Rubric §5.4.
- **agent-retry-outcome-correlation** — `BusEvent.AgentRetryScheduled(attempt, waitMs, reason)` fires when the retry is scheduled; the SUBSEQUENT `AgentRunStateChanged(Generating)` / `Failed` doesn't carry the retry's correlation. Ops can't answer "did retry #2 succeed?" without log-joining wall-clock timestamps. **方向：** thread a `retryAttempt: Int? = null` field on `AgentRunStateChanged` that echoes the attempt number the run is currently on (null when no retry yet). Pairs each subsequent state transition with the trigger. Rubric §5.4.
- **debt-eventrouter-runtime-test** — `apps/cli/.../EventRouter.kt` now subscribes to 6 bus event types (cycle 30's count); zero runtime test. Adding a 7th subscription silently trips no regression guard. Non-critical path per R.5 #10 so default P1. **方向：** new `EventRouterTest` constructs bus + stub renderer, publishes each event type, asserts the right renderer method fires with the right payload. Covers the routing pattern; no new Renderer behaviour coverage (that's RendererTest's lane). Rubric §5.6.
- **aigc-per-job-retry-cap** — `GenerateImageTool` / `GenerateMusicTool` / `GenerateVideoTool` / `UpscaleAssetTool` retry transiently-failed provider calls without a cap. A hung or hostile provider can chew through the session's spend budget silently. **方向：** cap retries per-job-id at N (default 3); on exhaustion surface `BusEvent.AigcJobExhausted(jobId, toolId, attempts, lastError)` + return a `ToolResult.Failed` with a clear "tried N times, giving up" message. Rubric §5.4.
- **debt-compactor-concurrent-process-audit** — `Compactor.process()` is `suspend` and could theoretically be invoked concurrently for the same session (manual `/compact` during auto-compaction trigger). `SessionStore` writes are per-call under mutex but the prune-then-summary-then-upsertPart sequence isn't atomic as a whole; two concurrent passes could double-drop + double-append compaction parts. **方向：** per-session mutex on `process()` (reuse or add to `Compactor` state). Test: fire two `process()` coroutines concurrently on the same session; assert exactly one `Result.Compacted` wins + exactly one compaction part ends up in the session. Rubric §5.6.

## P2 — 记债 / 观望

- **bundle-mobile-document-picker** — Android / iOS 当前限制于 app sandbox 内的 bundle (`<filesDir>/projects/` / `Documents/projects/`)。用户没法从 SAF / Files.app 选一个外部 bundle 打开。**方向：** Android 接 `Storage Access Framework` (`Intent.ACTION_OPEN_DOCUMENT_TREE`)，iOS 接 `UIDocumentPickerViewController`，结果 URI / NSURL 通过 platform-specific resolver 转成 Okio Path 喂给 `FileProjectStore.openAt`。**触发条件：** CLAUDE.md 平台优先级窗口 mobile 从"不退化底线"升级为主动开发。Rubric §5.4 / mobile。
- **bundle-talevia-json-split** — 当前 `talevia.json` 把 timeline + assets + source DAG + lockfile + snapshots 全装一个文件，单个 mutation 的 git diff 涨几百行；snapshot 多了文件可能涨到 MB 级。**方向：** 拆 `assets.json` / `timeline.json` / `lockfile.json` / `snapshots/<id>.json` 子文件；envelope `talevia.json` 只留 schemaVersion + 元数据 + 子文件清单。**触发条件：** 出现实际用户项目超限 diff 噪声 / snapshot 文件 ≥ 1 MB 的报告。Rubric §3a-3。
- **recents-registry-list-summaries-index** — `RecentsRegistry.listSummaries()` 当前每次扫所有 entry + 解码每个 `talevia.json` envelope，N≥几百时变慢。**方向：** registry 自身缓存 `(title, updatedAtEpochMs)`，envelope 写时同步 registry；`listSummaries` 直接读 registry 不再扫 bundle。**触发条件：** 有实测 profiling 数据支持 N≥几百时 `listSummaries` 成为瓶颈。Rubric §5.3。
- **debt-bundle-fs-testkit-copy-recursive** — Okio 的 `FakeFileSystem` 没有递归 copy 原语；`BundleCrossMachineExportSmokeTest` 在测试里手写了 `copyDirectoryRecursive(fs, src, dst)`（12 行）。**方向：** 第二个 bundle-level 测试需要同一个 walker 时，抽到 `ProjectStoreTestKit` 或 `BundleFsTestKit`；别提前做（don't pre-abstract on N=1）。**触发条件：** 出现第二个 caller。Rubric 外 / 顺手记录。
- **debt-register-tool-script** — 加一个新 tool 要同步改 4 个 `AppContainer`（CLI / Desktop / Server / Android；iOS 不注册 tool），每次 import 行手插入位置错了就挨 ktlint 一顿。P1 的 `debt-cross-container-tool-list-builder` 若落地则本条自动废弃；若那条先被阻塞，script 也是可选 workaround。**方向：** 脚本 `/register-tool <ToolName> <ctorArgs>` 生成跨 4 文件的 diff + 自动跑 `ktlintFormat`。**触发条件：** `debt-cross-container-tool-list-builder` 被证明不可行 且 连续 10 个新-tool cycle 都吃这份税。Rubric 外。
- **debt-unified-dispatcher-select-plugin-shape** — 每次给 `project_query` / `session_query` / `source_query` 加一个 select 要改 7-8 个协调站点（SELECT_* 常量 + ALL_SELECTS + Input 字段 + helpText + JSON Schema + `rejectIncompatibleFilters` 矩阵 + execute 分派 + sibling 文件）；reject 矩阵是 `O(n_selects × n_filters)` 成本。**方向：** 一个 select 一个文件的 "select plugin" shape（例：`@AppliesTo(SELECT_FOO)` 注解驱动 filter 适用性），把 7-8 站点收敛为 1 站点。**触发条件：** 任一 dispatcher 的 select 数达到 20，或 `rejectIncompatibleFilters` 矩阵达到 30+ 规则。当前：project 14 / session 15 / source 5 / provider 2。Rubric 外。
- **debt-desktop-missing-asset-banner** — cycle-3 落地了 CLI 侧的 `BusEvent.AssetsMissing` 警告（见 `2026-04-23-debt-export-missing-asset-warning.md`），但"Desktop 在 SnapshotPanel / ExportPanel 旁边加 banner" 仍未落地。**方向：** Desktop AppContainer 订阅同一个 bus 事件；SnapshotPanel / ExportPanel 附近加一个 warning banner，内容同 CLI 的 notice（count + 最多 5 条路径 + 溢出尾注）。**触发条件：** Desktop 达到 CLI parity 线后（CLAUDE.md 平台优先级）。Rubric 外。
- **source-node-history-query** — no way to see "how did this source node's body evolve?" — `update_source_node_body` overwrites silently, losing drafts. Git log is the only audit trail and loses bundle-internal semantic diffs. **方向：** `source_query(select=history, nodeId=X)` returns past body snapshots from a bounded ring buffer (say last 20 body mutations per node, persisted inline on the SourceNode as an append-only `bodyHistory: List<BodyRevision>`). `§3a #3` says "append-only list on Project blob needs its own SQLDelight table" — but SourceNode.body is stored inline anyway; 20-revision ring buffer on each node is bounded. Alternative: one-file-per-node history in the bundle (`<bundle>/source-history/<nodeId>.jsonl`). Decision cycle first. Rubric §5.5.
