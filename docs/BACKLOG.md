# Backlog

`/iterate-gap` 的驱动源。每个 cycle 的流程：

1. 读这个文件，挑**最靠上的未完成**任务（P0 → P1 → P2，同档内按出现顺序）。
2. plan → 实现 → 验证 → 归档决策 → commit + push。
3. 在 iteration 的 commit 里顺手把本任务的 bullet 从这个文件里删掉（不是打钩，直接删），保持列表紧凑。
4. 如果这个文件的 P0/P1/P2 三档都已清空（或文件不存在），`/iterate-gap` 会按 skill 里的 rubric 分析一次性生成 **20 条新任务**写回这个文件、commit，然后**在同一个 cycle 里继续**挑新生成列表里第 1 条开干。

任务描述只给 **Gap（现状 / 痛点）** 和 **方向（期望动的东西）** —— 细节留给 cycle 里的 plan 阶段现读代码决定。Rubric 轴用 `§5.x` 标。

## P1 — 中优，做完 P0 再排

- **aigc-result-multi-variant** — `generate_image` / `generate_video` 每次产 1 个 asset；OpenAI / Replicate 都支持 `n` 参数返 N 个候选让人挑。当前 agent 拿不到。**方向：** AIGC tools 接收 `n: Int = 1`，返回 List<assetId>；lockfile 每个变体一条 entry，共享 inputHash + 不同 variantIndex。Permission ASK 一次性覆盖 N 次成本。Rubric §5.2。Milestone §later. · skipped 2026-04-25: §3a #4 + design ambiguity — bullet's "shared inputHash" semantic conflicts with seed-in-hash invariant (variants need different seeds → different hashes), AND video providers (Sora 2 / Veo) don't natively support n so per-variant calls have separate-billing UX implications the user should approve before shipping.
- **effect-kind-registry-cross-engine-tax** — §5.2 compiler 轴信号：每加一个 filter / transition / overlay kind，要在 FFmpeg + Media3 + AVFoundation 三套 engine 各写一遍 (CLAUDE.md "Known incomplete" 已确认 vignette / transitions / subtitles / LUT 这套对齐税)；下一个新 kind 的 cross-engine 落地成本是 `≥ 3 platforms × ≥ 1 file each`。**方向：** core 层提供 `VideoEffectKind` 注册表 + `EffectHandler<TConfig>` 接口，每个 engine 实现一次 dispatch loop 而不是 per-kind switch；core 一处加 kind 后，三 engine 实现"找不到 kind" → fail loud 的同质 fallback。把跨平台对齐从手工责任翻成 compiler 检查到的接口契约。Rubric §5.2 / §5.6。Milestone §later.
- **session-spend-hard-cap-enforcement** — M2 exit summary §2 标记的 gap：`Session.spendCapCents` 已存在 (cycle 124)，但 `Agent.run` 不会在 AIGC tool 分发前 inspect "this session spent > cap → refuse with permission ASK"；spend cap 是名义存在的 reading 接口，不是写入接口的强制边界。**方向：** 在 AigcBudgetGuard 里加 cap-vs-`session_query(select=spend_summary)` 比较；超过则提升到 `aigc.budget` ASK 而非 `aigc.generate`，让用户能看到剩余预算并明确允许超支。Rubric §5.7 / §5.4。Milestone §later.
- **tool-discovery-meta-tool** — §5.4 expert path 信号：今天 LLM 知道有哪些 tool 是因为它们在 spec 里固定铺设；用户从 CLI / Desktop 想问"agent 现在能用哪些工具" 没有 query 通路（只有 `/help` 列 slash 命令，不列 LLM tool）。**方向：** `tool_query(select=catalog | spec_for_id, filter=...)` —— 暴露 tool registry 自身作为可读对象；用 `id` filter 让 agent 在 dispatch 前自查"我用过 set_clip_position 吗"或在 multi-step plan 里 reference 工具能力。OpenCode 的 `/system tools` 就是这条 spec。Rubric §5.4 / §5.2。Milestone §later.
- **desktop-merge-conflict-strategy** — §5.4 multi-user 信号：bundle 已 git-friendly，`talevia.json` 是 pretty-printed JSON，但两人同时改相邻 clip / 同一段 source node body，git merge 会产生文本冲突，用户要手撸 JSON。**方向：** 提供 `talevia merge --base <ref> --ours <ref> --theirs <ref>` 命令，理解 timeline / source / lockfile 三块的合并语义（timeline = clip-id-keyed merge；source = node-id 替换冲突探测；lockfile = 永远 union）。或最少做出 `git merge` driver 在 `.gitattributes` 里注册成 `merge=talevia-json`。Rubric §5.4 / §3.4。Milestone §later.
- **debt-tool-cluster-source-query-overlap** — R.5 #2 scan：`source_query` / `source_node_action` / `find_stale_clips` / `consistency_propagation` 等 source-side 操作在 7+ 工具间分散；近似群信号触发（`source_query(select=...)` 已 18+ select，再加新 source 工具会让 agent context tax 直接膨胀）。**方向：** 在 plan 阶段 grep 类似前缀工具 / select 重叠度，candidate 合并：source mutation tools 全部 fold 到 `source_node_action` 既有 dispatcher（cycle 113 已起）；剩下不易 fold 的拿去 `tool_query(select=catalog)`。Rubric §5.6 / §3a-1。Milestone §later.
- **debt-add-runtime-test-replay-lockfile-flow** — R.5 #10 scan：`ReplayLockfileTool` 这条 critical-path（M2 criterion 6 验证用）有 ReplayLockfileToolTest 覆盖 happy path，但 grep `ReplayLockfileTool.execute` 在 e2e 路径只在 `RefactorLoopE2ETest` 调用一次；replay 路径的 cross-tool 健全度依赖 e2e 对 deterministic provider 的 mock，没有覆盖"replay → cache invalidation race"或"replay 在并发 mutate 下"。**方向：** 加一个 e2e test：concurrent agent.run() + replay 同 inputHash，断言两个不会撞在同一 lockfile entry / 不会丢 replay 产物。Rubric §5.6 / §5.3。Milestone §later.
- **debt-flaky-provider-warmup-test** — `ProviderWarmupKickoffTest.successfulProviderEmitsStartingThenReady` 在 cycle 115 偶发 5s 超时（passed on retry）；cycle 127 没观察到新 flake，但 trigger 仍是"再观察 ≥ 1 次 flake"。**方向：** 切到 `runTest { ... }` 内 test dispatcher，用 `advanceUntilIdle` 替换 yield-loop；或加 `awaitReady` 屏障 (类似 `BusEventTraceRecorder.awaitReady`)。**触发条件：** 再观察 ≥ 1 次 flake，或下次 repopulate 仍未自然消除（已经 13 cycles）—— 升 P0 修。Rubric 外 / 顺手记录。Milestone §later.
