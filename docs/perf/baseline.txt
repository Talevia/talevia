# Talevia perf baseline

Captured by `./gradlew :core:bench` on a single dev machine (M-series Mac).
Each line: `<benchmark-name> elapsed=<wall-time> softBudget=<budget>`.

Numbers are advisory. The benchmarks themselves never `assertTrue` against
absolute wall-time (per the cycle-2 budget policy in each `*Benchmark.kt`'s
KDoc) — they print, the reviewer eyeballs `over` markers, and a
2× regression triggers a follow-up cycle. CI runs `:core:bench` on every
PR (see `.github/workflows/ci.yml`); the printed lines accumulate in CI
logs as the perf history curve.

## How to refresh

```
./gradlew :core:bench 2>&1 | grep '\[bench\]'
```

Update the table above when the wall-time deltas drift > 25% in a single
cycle (regression OR improvement worth pinning). The soft-budget column
matches each benchmark's own `softBudget` constant — change those at the
benchmark's source if real perf shifts make them wrong.

## What's NOT here

- `BusEventPublishBenchmark` — uses `assertTrue` against an internal budget
  rather than printing a `[bench]` line. It still runs under `:core:bench`
  (the filter `io.talevia.core.bus.BusEventPublishBenchmark` matches it);
  pass/fail is the signal.
- Memory baselines — the v1 perf gate is wall-time only. Memory regression
  tracking is a follow-up if heap profiles become a recurring pain point.
- Cross-machine variance — these numbers are from one dev machine. CI's
  `ubuntu-latest` will print different absolute numbers; the *shape* of
  the curve (which benchmarks moved how much) is what matters.
