## 2026-04-22 — Delete MediaStorage / FileMediaStorage / InMemoryMediaStorage / FileBlobWriter (VISION §3a-3 rubric axis, design-constraint §3a-1 "no tool/interface duplication")

**Context.** Post-`baad43f` file-bundle migration replaced the global media catalogue + machine-wide blob directory with `Project.assets` + `BundleMediaPathResolver` + `BundleBlobWriter`. But six core tools (`AddClipTool` / `ReplaceClipTool` / `TrimClipTool` / `ApplyLutTool` / `ExtractFrameTool` / `ImportMediaTool`'s reference-import path) plus three AIGC/ML tools (`GenerateImageTool` / `GenerateVideoTool` / `UpscaleAssetTool` — character-ref path resolution) plus three "consume a resolver" tools (`DescribeAssetTool` / `TranscribeAssetTool` / `AutoSubtitleClipTool`) still received a `MediaStorage` constructor arg, and every one of the five `AppContainer`s had to build a `media: MediaStorage = InMemoryMediaStorage()` placeholder + a `blobWriter = FileBlobWriter(<tmpdir>/talevia-generated)` to satisfy those deps. Two parallel paths for the same responsibility — rubric delta §3a-3 (duplicate interface / duplicate catalogue) **有 → 无**.

**Decision.** Delete the legacy interface + every implementation + every consumer:
- Drop `core/platform/MediaStorage.kt`, `InMemoryMediaStorage.kt`, `MediaBlobWriter.kt` (common), `jvmMain/FileMediaStorage.kt`, `jvmMain/FileBlobWriter.kt`, `apps/android/.../AndroidPersistentMediaStorage.kt`, `apps/android/.../AndroidFileBlobWriter.kt`, `apps/ios/Talevia/Platform/IosFileBlobWriter.swift`, `core/jvmTest/.../FileMediaStorageTest.kt`, the `importWithKnownMetadata` Swift bridge in `IosBridges.kt`.
- Clip-side tools (`AddClipTool` / `ReplaceClipTool` / `TrimClipTool` / `ApplyLutTool`) lose the `media: MediaStorage` param; asset-existence checks move inside `store.mutate { project -> project.assets.any { ... } }`.
- `ExtractFrameTool` signature changes from `(engine, storage, blobWriter: MediaBlobWriter)` → `(engine, projects: ProjectStore, bundleBlobWriter: BundleBlobWriter)`; it mints an `AssetId`, calls `engine.thumbnail`, writes via `BundleBlobWriter.writeBlob(pid, assetId, bytes, "png")`, and appends the new `MediaAsset` to `project.assets`. Output now travels with the bundle (was `<tmpdir>/talevia-generated/<uuid>.png`).
- `ImportMediaTool` drops `storage: MediaStorage`; the `copy_into_bundle=false` branch builds the `MediaAsset` inline (mint id, probe, construct `MediaSource.File(path)`) instead of round-tripping through `storage.import`. Bundle-copy branch unchanged.
- `GenerateImageTool` / `GenerateVideoTool` / `UpscaleAssetTool` drop the `storage: MediaStorage` / `resolver: MediaPathResolver` constructor args and build `BundleMediaPathResolver(project, bundleRoot)` on demand to resolve reference assets.
- `DescribeAssetTool` / `TranscribeAssetTool` gain an optional `projectId` input + `RequiresProjectBinding` applicability; they build `BundleMediaPathResolver` from the session's current project. `AutoSubtitleClipTool` already took a `ProjectStore`, just drops its `resolver` param.
- `ProxyGenerator` contract changes from `generate(asset)` → `generate(asset, sourcePath)`; callers are responsible for resolving `MediaAsset.source` → absolute path before invoking (trivial for `MediaSource.File`; for bundle-local assets the caller knows the just-written path). `FfmpegProxyGenerator` / `Media3ProxyGenerator` / `AVFoundationProxyGenerator` drop their `MediaPathResolver` constructor args — they receive the path directly.
- Desktop `Main.kt:resolveOpenablePath` (tool-output → openable filesystem path, used by ChatPanel) now scans `container.projects.listSummaries()` to find the owning project, then resolves via `BundleMediaPathResolver`. Prefers the `projectId` field from the tool output when present for O(1) lookup.
- Desktop `Main.kt` manual import button + `TimelinePanel.kt` LUT-picker flow construct the `MediaAsset` inline and append via `store.mutate`, matching the tool-side pattern.
- Server `POST /media` now requires a `projectId` query param (400 when missing), probes metadata, copies bytes via `bundleBlobWriter.writeBlob`, and appends a bundle-file asset to `project.assets`. The old "shove bytes into a global catalogue without any project context" behaviour was already load-bearing-on-nothing after `baad43f` — bytes landed in the in-memory storage that died with the process.

**Alternatives considered.**
1. *Keep `MediaStorage` as a deprecated shim that proxies to `Project.assets`.* Rejected — the interface has four methods (`import` / `get` / `list` / `delete`) and none of them map cleanly onto a per-project inventory (`import` has no projectId; `delete` can't be safe because the same AssetId can be bound to clips across projects). Shim would either lie about semantics or re-derive the project from context the caller doesn't pass. Per memory `feedback_no_compat_clean_cuts.md`: default to deleting old code, no migration logic.
2. *Leave `MediaPathResolver` as a constructor arg on ML/upscale/autosubtitle tools and pass a project-dispatching resolver from the container.* Rejected — the container has no session context to pick "which project owns this asset", so the dispatcher would scan every project's assets on each resolve call. Moving `ProjectStore` into the tools' constructors and resolving per-call via `BundleMediaPathResolver(project, bundleRoot)` is the same shape the AIGC tools already use.
3. *Change `ProxyGenerator.generate` to take the full `MediaAsset` plus a per-call `MediaPathResolver` instead of a `sourcePath: String`.* Rejected — the caller (ImportMediaTool) already knows the path (just wrote it or received it from the user). Threading a resolver just to call `resolver.resolve(asset.id)` to recover a string the caller had in hand is ceremony. OpenCode's tool-dispatch reference (`tool/tool.ts`) passes resolved values, not resolver handles — behavior match.
