## 2026-04-23 — debt-server-container-env-defaults (VISION §3a-9 / 正确性)

**Context.** After `baad43f` replaced the SQL-backed project store with the file-bundle store, `ServerContainer.kt:215-218` started doing `env["TALEVIA_RECENTS_PATH"]!!.toPath()` / `env["TALEVIA_PROJECTS_HOME"]!!.toPath()`. Production (`Main.kt`) wraps `System.getenv()` with `serverEnvWithDefaults()` before passing it in, but every server test calls `ServerContainer(env = emptyMap())`. Result: 15 tests across `ServerSmokeTest` / `LifecycleTest` / `MetricsEndpointTest` / `InputValidationTest` NPE'd on clean `main` for 7+ iterate-gap cycles. Downstream pain point `debt-server-tests-externality` documented that every cycle touching server-adjacent code had to `git stash` + re-run `:apps:server:test` just to confirm "the red isn't mine" — a cycle-level externality the skill was paying over and over. Rubric delta §3a-9: server test suite 红 → 绿 (正确性 "部分 → 有" on CI signal for apps/server).

A second break turned out to be gating on the first: `ServerModule.kt` has two exhaustive `when` statements over `BusEvent` (`eventName` and `BusEventDto.from`) that cycle 14 forgot to update when it added `BusEvent.AssetsMissing`. Core's `Metrics.kt` got the exhaustive-when update; Server's did not. Both had to land in this commit for the test suite to turn green — the server tests couldn't even compile before the fix.

**Decision.**
1. **Moved `serverEnvWithDefaults` into `ServerContainer` itself.** The helper (now `withServerDefaults(raw: Map<String, String>): Map<String, String>` in `Main.kt`) takes a caller-supplied map, fills missing `TALEVIA_PROJECTS_HOME` / `TALEVIA_RECENTS_PATH` from `TALEVIA_DATA_DIR` or `~/.talevia`, and returns the filled map. `ServerContainer`'s constructor applies it unconditionally — production and tests share the same default-fill path.
2. **Renamed the ctor param from `env` to `rawEnv`** so the internal `private val env: Map<String, String> = withServerDefaults(rawEnv)` property can legally shadow it. Kotlin resolves the `env` identifier inside later property initialisers (`env["TALEVIA_RECENTS_PATH"]!!`) to a same-named primary-constructor parameter if one exists, which would silently bypass our default-fill. Using a distinct ctor param name closes that resolution hole. 13 test call sites updated from `ServerContainer(env = ...)` to `ServerContainer(rawEnv = ...)`.
3. **Main.kt simplifies to `ServerContainer()` default-construction.** The env is `System.getenv()` by default, the container fills bundle-path defaults, and no one needs to remember to wrap anything.
4. **`ServerModule.kt` exhaustive-when fixed.** Added `is BusEvent.AssetsMissing -> "project.assets.missing"` to `eventName`, and a `BusEventDto.from` case carrying `projectId` (AssetsMissing is not session-scoped). This brings the server's event-naming in parity with `core.metrics.Metrics` which cycle 14 already covered.

**Alternatives considered.**
- *Keep `serverEnvWithDefaults` in Main.kt and require every test to call it explicitly.* Rejected: the failure mode (forgetting the wrapper → NPE) is exactly what just bit us. Test ergonomics matter — every test-harness requirement to "remember step X before step Y" eventually rots.
- *Keep the ctor param named `env` and try a `init {}` block to reassign.* Rejected: `init` blocks can't reassign `val` properties, and if we made `env` a `var` we'd lose the "defaults applied once at construction" invariant. A rename is one line per test and permanent.
- *Pass `TALEVIA_DATA_DIR=<tempdir>` in every test so defaults resolve into a scratch dir.* Rejected on two grounds: (a) still requires each test to remember something; (b) the tests in question don't exercise `FileProjectStore.createAt` / `mutate` — they just spin up the container and hit HTTP endpoints. `FileProjectStore`'s constructor is zero-IO (only path storage; mkdirs are lazy), so pointing defaults at `~/.talevia/projects` is safe for the non-bundle-write tests and doesn't pollute `$HOME`. Tests that do need a scratch dir can still pass `rawEnv = mapOf("TALEVIA_DATA_DIR" to "...")`.

**Coverage.** `./gradlew :apps:server:test` now passes all 15 tests that were red on clean `main`: `ServerSmokeTest` (8 methods including `authTokenRequiredWhenEnvSet` that specifically tests env-var handling), `LifecycleTest` (2 methods), `MetricsEndpointTest` (3 methods), `InputValidationTest` (4 methods). No new tests added — the bug was that 15 existing tests were failing, so turning them green IS the coverage signal.

**Registration.** 3 files under `apps/server/src/main` + 4 files under `apps/server/src/test` (no behavioural changes, just ctor-arg rename). No new `Tool<I, O>`, no AppContainer sweep. `ServerModule.kt`'s BusEvent-exhaustiveness fix is local to the server module — Core's `Metrics.kt` was already current.
