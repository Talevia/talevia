## 2026-04-23 — bundle-cross-process-file-lock (VISION §5.3 / 正确性)

**Context.** `FileProjectStore` serialises writes within one process via `kotlinx.coroutines.sync.Mutex`, but has no cross-process exclusion. A developer running CLI + Desktop + Server simultaneously against the same `~/.talevia/projects/<id>/` bundle can hit the classic lost-update pattern: process A reads `talevia.json`, process B reads the same snapshot, both apply their own mutation, both `atomicMove` the rewritten envelope — the second write silently overwrites the first. The old SQL-backed store got away without explicit cross-process locking because SQLite's `BEGIN IMMEDIATE` transactions handled it; the file-based bundle has no such built-in guard. Rubric delta §5.3 / 正确性 (multi-process bundle access): 无 → 有 (fail-loud on contended writes).

**Decision.** Added a `BundleLocker` interface in `core.platform` (commonMain) with three members: `tryAcquire(Path): Handle?`, `Handle.release()`, and a `BundleLocker.Noop` default. `JvmBundleLocker` in `jvmMain` implements `tryAcquire` via `java.io.RandomAccessFile` + `FileChannel.tryLock()` on `<bundle>/.talevia-cache/.lock` (gitignored, machine-local per the existing cache-dir convention). `FileProjectStore`'s ctor takes `locker: BundleLocker = BundleLocker.Noop` and wraps every write-path (`createAt`, `upsert`, `setTitle`, `mutate`, `delete`) with a `withBundleLock(path) { ... }` helper that fails loud with `IllegalStateException("Bundle is locked by another Talevia process: $path")` when `tryAcquire` returns null. CLI / Desktop / Server containers inject `JvmBundleLocker()`; Android / iOS stay on `Noop` (single-app sandbox → single process per bundle, so OS-level file locks are cost without benefit there).

Lock scope is "any write path", not "any read path". Reads (`get` / `openAt` / `summary` / `listSummaries` / `list`) don't acquire the lock because our writes use atomic rename (`okio.FileSystem.atomicMove`), so a reader always sees either the pre-write or post-write envelope — never a torn read. Locking reads would serialise them against writes at no correctness gain.

**Alternatives considered.**
- *`expect fun tryAcquireBundleLock(lockFile: Path): Handle?` instead of an injected interface.* Rejected: testability — the concurrency test needs to observe the `null` branch with two locker instances in one JVM, and an `expect` function can't be stubbed. An interface+DI is how the rest of `core.platform` (`VideoEngine`, `MediaPathResolver`, `SecretStore`) already does platform abstraction; consistency wins.
- *Sidecar `.lock` file at the bundle root (not under `.talevia-cache/`).* Rejected: the root of the bundle is the user's git-tracked area. A stray `.lock` file leaking into `git status` every time a crashed process left one behind is exactly the friction we don't want. `.talevia-cache/` is already gitignored (via the auto-written `.gitignore` on bundle create), so the lock file is invisible to users.
- *SQLite-as-lock (open an in-memory DB and use `BEGIN IMMEDIATE`).* Rejected: re-adds a SQL dependency to the write path we just deleted in `baad43f` when removing `SqlDelightProjectStore`. Using `FileChannel.tryLock` on a one-byte sidecar matches the OS-level primitives (advisory `flock` on POSIX, `LockFileEx` on Windows) directly — this is the standard approach in git's own `packed-refs.lock`, `node_modules/.package-lock`, and the `java.nio.channels.FileLock` convention.
- *Retry / backoff when `tryAcquire` fails.* Rejected per the bullet's "fail-loud" guidance: silent retries hide bugs. If the user genuinely wants serialisation, close the other Talevia instance. A future CLI / UI could wrap the `IllegalStateException` with a "close the other Talevia process?" prompt; not doing that here because the lock-contention path must be observable, not automagically worked around.

**Coverage.** `core/src/jvmTest/kotlin/io/talevia/core/domain/FileProjectStoreFileLockTest.kt`:
- `secondLockerFailsLoudWhenFirstHoldsLock` — two `JvmBundleLocker` instances in one JVM; first holds the sidecar lock, second tries `mutate`, expects the "locked by another Talevia process" `IllegalStateException`, then after `release()` the second `mutate` succeeds. `FileChannel.tryLock`'s `OverlappingFileLockException` path in `JvmBundleLocker` simulates a second process in production.
- `noopLockerNeverFailsEvenUnderContention` — verifies `BundleLocker.Noop` ignores an externally-held OS lock. Protects the Android / iOS / in-memory-test contract.
- `lockReleasedAfterSuccessfulWrite` — 5 sequential `mutate`s on one store. If `withBundleLock` leaked handles between calls, the second `mutate` would fail-loud. Guards the finally-release invariant on the happy path.
- `lockReleasedEvenWhenMutateBlockThrows` — a tool throws mid-mutation; subsequent `mutate` must still succeed. Guards the finally-release invariant on the failure path. Without this, a single bad tool call would "stick" the bundle until restart.

**Registration.** 3 JVM containers touched: `apps/cli/.../CliContainer.kt`, `apps/desktop/.../AppContainer.kt`, `apps/server/.../ServerContainer.kt` — each adds one import + one ctor arg. `apps/android/.../AndroidAppContainer.kt` + `apps/ios/.../AppContainer.swift` unchanged: default `BundleLocker.Noop` fits their single-app-sandbox model. No new `Tool<I, O>` → no tool-registration sweep required.
